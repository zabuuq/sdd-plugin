## /prd
- **Session paused partway through.** PRD interview started but not completed. Resumption state saved to `session-state.md`.
- **Onboarding decision:** Jason decided `/onboard` should be a one-time-per-machine step, not per-project. User profile stored globally. `/scope` checks for it and gates on it.
- **Scope sizing:** The `/scope` phase should evaluate project size and recommend breaking large projects into independent bite-sized pieces, each going through the full SDD workflow. Aligns with microservices thinking.
- **Backlog/transition flow (partially explored):** Backlog items captured during planning, surfaced during `/iterate`. If an item deserves its own project, the system offers to close the current project (with `/reflect`), seed a new one with context, and start fresh. Jason wants this fully speced out with detailed questions — was interrupted before completing.
- **Active shaping:** Jason corrected the agent's framing early, pointing out the plugin is an adaptation of the hackathon plugin (not built from scratch) and that the scope should be based on that reference implementation. Drove the backlog/transition concept himself and explicitly asked for it to be deeply explored.
- **Backlog detail level:** Jason wants maximum information per entry — what it is, why deferred, what it connects to, constraints discussed. Reasoning: makes future scoping easier, and "I'm getting old and I forget things."
- **Backlog location:** `docs/backlog.md` in the project directory. Most backlog items will be continuations of the current project. For rare cases needing a new project, a seed file is generated and copied over.
- **Sprint concept emerged:** Jason worked through the tension of core vs. nice-to-have without time-boxing and arrived at a sprint-based workflow. The conversation evolved from "do we need sprints?" to "sprints replace checklist" to a full sprint loop with refinement. This was entirely Jason-driven — he proposed the sprint idea, worked out the loop structure, and identified the need for `/refine` as a separate command.
- **Command chain changed significantly from scope doc:** Ten commands total. `/sprint` replaces `/checklist`. `/refine` is new. `/retro` is the project-level close. `/plugin` captures improvement feedback. `/onboard` is one-time per machine with git preference, communication style, and workflow explanation.
- **Partial story splitting:** Jason identified the problem of partially completed PRD stories across sprints. Decided on splitting during `/reflect` — completed work becomes one story (checked off), remaining work becomes another (still open).
- **Testing strategy:** Lives in `/spec`. Applied per-item in `/build`. Automated or manual verification based on what the spec defines.
- **Tone is user-configurable:** Not hardcoded. Captured during `/onboard` as communication style preference. Presented as multiple choice with an "other" option and leading examples.
- **CLAUDE.md and AGENTS.md:** AGENTS.md is agent-agnostic (architecture, conventions, patterns). CLAUDE.md is Claude-specific with explicit instruction to read AGENTS.md. Both generated in `/spec`, both living documents. `/build` loads sprint context itself — CLAUDE.md doesn't point at sprint files.
- **Command explanations:** Each command outputs a brief explanation of its purpose the first time it runs in a project. Not gated, just text. Tracked in a per-project state file.
- **Markdown-first tracking:** No GitHub issues/milestones for v1. Git repos supported based on onboard preference, but issue tracking stays in markdown files.
- **Sprint sizing:** No time-boxing. Grouped by dependency, relatedness, logical sequence. Each sprint should end with something tangible the user can see, use, or test. Plugin is opinionated about grouping but user makes final call.
- **Process notes split by sprint:** Planning phases share one file, each sprint cycle gets its own.
- **Living documents with resistance:** PRD, spec, CLAUDE.md, AGENTS.md are all updatable, but the plugin's default is to resist changes and push new items to backlog or unvetted. Re-scoping is framed as a weighty decision.
- **Deepening rounds:** Two rounds beyond mandatory questions. Round 1 surfaced: sprint sizing philosophy (tangible milestones over time-boxing), `/refine` updating spec with user agreement during conversation, CLAUDE.md not referencing sprint files, command explanation blurbs. Round 2 surfaced: `/onboard` explanation as scrollable text not gated, plugin judgment on iteration sizing, `/plugin` as the improvement feedback command name.

## /spec
- **Technical decisions and rationale:**
  - Pure markdown skills architecture — no runtime code. Matches reference implementation, zero dependencies. The plugin is its own specification.
  - Marketplace repo structure from day one — Jason plans to share with friends, colleagues, and back to the Devpost hackathon community.
  - Plugin namespace: `sdd`. Commands become `/sdd:onboard`, `/sdd:scope`, etc. Avoids collisions with built-in `/plugin` command.
  - `/plugin` command renamed to `/feedback` — clearer name, avoids collision with Claude Code's built-in `/plugin` command. Jason flagged this himself.
  - sdd-guide split into core SKILL.md + reference files (deepening-rounds, context-loading, living-documents, open-concerns) to reduce context overhead per command.
  - Surgical context loading for `/build` — only loads the specific spec subsection and PRD story for the current checklist item, not full documents. Directly informed by Jason's experience hitting auto-compaction on his other hackathon project.
  - `docs/open-concerns.md` as a formal cross-phase concern tracking mechanism. Motivated by Jason discovering that △ feedback from /scope and /prd sessions wasn't carried forward — concerns evaporated with the conversation.
  - PRD health monitoring: 10+ epics → recommend phase-splitting, 5+ unvetted items → flag scope creep.
  - `/refine` updates documents incrementally in dependency order (PRD → spec → AGENTS.md → CLAUDE.md), showing each change to user before proceeding.
  - `/feedback` auto-captures context (timestamp, sprint number, last command, project directory) so the user's note has full traceability during `/retro`.
  - `project-state.json` tracks `lastCommand` field — each command updates it on startup. Enables `/feedback` context capture.
  - Lightweight validation (Option C): validate binary pass/fail conditions (JSON parses, files exist, sprint has unchecked items) but trust the agent for fuzzy cross-reference resolution.
  - Global user profile at `~/.claude/sdd-user-profile.json` — accessible from any project directory.
  - Each command skill explicitly loads sdd-guide core via path reference (deterministic, not auto-discovery) — mirrors the proven hackathon plugin pattern.
- **What Jason was confident about vs uncertain:**
  - Confident: pure markdown architecture, marketplace structure, file structure layout, open-concerns mechanism, context loading approach, incremental refine updates, feedback context capture.
  - Drove the decisions on: renaming `/plugin` to `/feedback`, PRD thresholds (rejected line count, chose epic count + unvetted count), incremental updates over batch, and the need for open-concerns based on real gaps he discovered across sessions.
  - Deferred to agent recommendation on: open-concerns file format (Option 1 — dedicated file), validation strategy (Option C — lightweight), sdd-guide referencing approach (Option A — explicit path), sdd-guide splitting into core + references. In each case, asked for pros/cons before deciding.
- **Stack choices:** No traditional stack — pure markdown skills plugin. Same architecture as the hackathon-in-a-plugin reference. No build step, no runtime, no dependencies.
- **Deepening rounds:** Two rounds beyond mandatory questions.
  - Round 1 surfaced: `docs/open-concerns.md` mechanism for cross-phase continuity (motivated by Jason finding lost △ items from previous sessions), PRD size thresholds (10+ epics, 5+ unvetted), `/refine` incremental update ordering (PRD → spec → AGENTS.md → CLAUDE.md with user confirmation at each step), `project-state.json` format (JSON).
  - Round 2 surfaced: `/build` re-entry context loading strategy and full load order specification, context limit mitigation through surgical loading (directly informed by Jason's auto-compaction experience on his other project), per-command loading rules table, sdd-guide split into core + references to minimize context overhead, `/feedback` auto-context capture (timestamp, sprint, last command, project dir), `lastCommand` field added to project-state.json, lightweight validation strategy (binary pass/fail only, trust agent for fuzzy matching), architecture self-review identifying sdd-guide size as a concern.
- **Active shaping:** Jason drove the direction throughout. Key moments: (1) He discovered the open-concerns need himself by digging through previous session outputs and finding lost △ feedback — this wasn't prompted by the agent. (2) He flagged the `/plugin` naming collision with Claude Code's built-in command before the agent caught it. (3) He rejected line count as a PRD threshold metric on his own reasoning ("too arbitrary, not easy to enforce") and proposed the combination approach. (4) He interrupted the deepening round to surface the context limits topic that the agent had mentioned but moved past. (5) He brought real evidence from his other hackathon project (auto-compaction experience) that directly shaped the context loading architecture. The agent followed his lead more often than the reverse — particularly on open-concerns and context management, which became the two most architecturally significant decisions of the spec phase.
