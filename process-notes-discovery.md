### Decision — replace, not coexist
- **Confirmed:** v6 is a **complete top-down replacement** of the current discovery→scope→prd→spec linear front end with the brainstorm-first flow — *not* a fast-lane that sits alongside the old chain.

The current linear chain is the thing that's stalling completion, not just slowing it. The replacement is motivated by *throughput of finished projects*, not just per-step speed. This deepens Beat 2: the cost isn't only the 8-hr doc slog in the abstract — it's that the slog is causing projects to die before they ship.

### Phase 2, Round 1 — accepted
- Resumed via `/sdd:unpause`; re-confirmed the locked Phase 1 picture, user accepted the one-round recommendation. Running the round (5 questions default, nature register). Topics: speed-vs-interview tension, success definition, prior-art tension, lessons-as-they-happen.
- **Q1 — speed-vs-interview tension.** User sharpened the root cause: the slog was **four separate interviews** (discovery/scope/prd/spec), each 5–20 questions, with two compounding failure modes — (a) **repetition** across commands, and (b) **cross-phase loss**: a spec-related thought raised during scope got deferred, then lost in the shuffle by the time its command arrived. Combining scope+PRD+spec into one drafted pass attacks both — fewer total questions, and cross-cutting points captured while fresh in the same doc set. Original split assumed one combined interview would be "too much info at once"; user now reads the split *itself* as the tax, not the volume. **The limit on interview bloat is structural (one unified interview vs. four blank-page ones), not a hard question cap.**
- **Q2 — success definition.** Two signals. (1) **Concrete milestone:** get from discovery to a **high-fidelity prototype in a single day**, at least once — never achieved with the current flow; not expected every time, but landing it once proves the front end is fast enough. Doubles as a whole-front-end stress test (brain-dump→3 docs→interview→research→prototype must all fit in a day). (2) **Felt state (also "very important"):** the front end didn't drain him and he still had momentum when he hit code — opposite of arriving at the build already spent.
- **Q3 — prior-art tension.** "The talk sparked it, but it's my own flow now." The six-step model is a **spark, not a spec** — no part of it is load-bearing on its own. Divergences (medium sprints, whole-project docs) are the flow being itself, not exceptions to a followed model. Framing for discovery doc: credit the talk as inspiration, don't measure v6 against it as a standard.
- **Q4 — lessons-as-they-happen (nature of a "lesson").** Taxonomy of what's worth catching, mostly **project-level**: (a) **lost-context / redundant-question events** — a later step re-asks something already answered (has happened multiple times; same cross-phase-loss failure from Q1, surfacing as a lesson); (b) **repeated roadblocks** — user or AI hits a wall, solves it, hits the same wall again; the solution must be recorded so it isn't re-solved; (c) **course corrections** — AI goes one way, user says "no, different direction"; the redirect is a lesson; (d) **reusable designs / code practices / preferences**. **Altitude routing:** project-level is the main target (lessons specific to a project, repeatable within it, live with the project); **global lessons** (true across projects, e.g. "I always use repos") should be recordable at a global level too, and higher-level *plugin* lessons flow to `/sdd:feedback`.
- **Q4 addendum — lessons feed the retro (fixes hollow retro).** The lessons captured as-they-happen ARE the retro's raw material. Retro flow: **replay the items marked as lessons during the project** for the user to confirm/acknowledge; if lessons were caught, surface them; otherwise (or after) ask if there's anything to bring up; if "no," the retro ends. This is the concrete fix for the hollow "yeah, everything was great" retro — the retro reviews pre-captured lessons rather than trying to discover them from a blank page at the end.
- **Q5 — the prototype's nature ("both or neither").** Not a throwaway mirror, not a code seed — a **durable reference artifact**: (a) used to compare against the 3 docs; (b) **client/investor-facing** — shown to real people, so it must be solid, presentable, and iterated on the user's feedback (it grows); (c) a **guide/spec-by-example for the final product**. Key constraint: **the prototype's CODE is disposable, but the prototype itself is durable.** Building the final product off the prototype's code "will probably lead to disaster" — the final product starts **fresh**, using the prototype as a north star, not a foundation. Design implication: invest in the prototype as a communication/reference tool; never let the real build inherit its HTML/CSS/JS.
